# agents/STRUCTURE.md
> Per-project structural map — auto-generated by `build-component` on first run.
> The plugin reads this file before any UI work to know where things live
> and which surfaces apply. Edit by hand only if your project structure
> changes (folder rename, new app added, surface dropped).
> Never duplicate this content into other canon — `build-component`,
> `design-check`, and `audit` all read from here.

---

## Surfaces present

Which user-facing surfaces this project ships. `no` rows are hidden from
tier prompts (e.g. a dashboard-only project never sees Marketing tier).

| Surface     | Present | Path                                  | Tech          |
|-------------|---------|---------------------------------------|---------------|
| Marketing   | yes/no  | e.g. frontend/src/app/(marketing)/    | Next.js 16    |
| Web app     | yes/no  | e.g. frontend/src/app/(dashboard)/    | Next.js 16    |
| Desktop app | yes/no  | e.g. desktop/src/                     | Tauri+React   |
| Docs        | yes/no  | e.g. frontend/src/app/docs/           | MDX           |
| Mobile      | yes/no  | n/a                                   | —             |

---

## Component locations

Where components live, split by tier. `build-component` writes new files
into the path matching the chosen tier — never elsewhere.

| Tier          | Path                                    | Notes                    |
|---------------|-----------------------------------------|--------------------------|
| Generic       | frontend/src/components/ui/             | shadcn primitives        |
| Marketing     | frontend/src/components/marketing/      | only used by (marketing) |
| App (web)     | frontend/src/components/app/            | only used by (dashboard) |
| App (desktop) | desktop/src/components/                 | only used by desktop app |

---

## Stack per surface

- **Marketing / Web app**: Next.js 16, TypeScript, Tailwind v4, shadcn
- **Desktop app**: Tauri 2, React + TypeScript, Tailwind v4, shadcn (shared convention with web)
- **Docs**: MDX inside the Next.js app, same Tailwind config

---

## Conventions detected

Detected silently on first run by reading the existing codebase. Do not
change these unless the codebase itself changes — `build-component`
matches whatever's already here.

- **Style system**: Tailwind v4 + shadcn (cva + forwardRef + cn triplet for primitives)
- **File naming**: kebab-case (e.g. `button.tsx`, `pricing-card.tsx`)
- **Folder structure**: `components/ui/` for primitives, sibling folders for blocks
- **Token chain**: `agents/DESIGN.md` → `globals.css` (CSS vars) → Tailwind theme → component class
- **Icon library**: Lucide (one library, `currentColor`, never emoji)
- **Forms**: react-hook-form + zod
- **State**: React Query for server state, Zustand for client (if present)

---

## Cross-tier import rules (enforced by `audit`)

- Marketing components **cannot** import from `components/app/` (or wherever App tier lives)
- App components **cannot** import from `components/marketing/`
- Both can freely import Generic primitives from `components/ui/`
- Cross-surface sharing only via Generic tier — never sibling-imports
- Desktop and web App tiers are siblings: neither imports the other
