# project-protocol Audit — 2026-05-26

Scope: v2.4.0 source. All 15 skills, every reference file, every template, hooks.json, both manifests, README, CHANGELOG. Five questions per artifact + systemic checks.

---

## 1. Executive Summary

1. **TOOLING.md encodes Vish's personal path** (`~/Arel OS/Projects/`) verbatim in the "global standard" that every project receives — any non-Vish user gets a broken standard.
2. **design-check does not know it exists as a skill at build time** — its auto-chain relies on the host environment "auto-firing skills on description match," which is not a guaranteed Cowork/Claude Code behavior; the chain can silently drop.
3. **save-session Step 10 assumes a worktree branch, then merges to main** — projects working directly on `main` (common for solo users) skip the push and leave canon changes stranded.
4. **build-page's "Required canon" gate for marketing pages will halt on nearly every mid-project invocation** because `agents/marketing/layouts/<slug>.md` is listed as required but is the last file marketing-brief writes — briefs and copy exist before layouts do.
5. **No skill handles the "monorepo root" case** — `package.json` at root but code lives in `apps/web/` — meaning Node detection silently misfires, Bun rules inject into a non-Node workspace root, and STRUCTURE.md detection looks in the wrong folders.

**H1 — TOOLING.md contains a personal filesystem path baked into the "global standard"**

File: `templates/TOOLING.md`, line 5; `skills/init-project/references/phase-4-design-system.md`, lines 110–120.

The template explicitly references `~/Arel OS/Projects/` in the rule "Never run `pnpm install` … in any project under `~/Arel OS/Projects/`." This string ships verbatim into every project Vish initializes — but it's also the template copied to `agents/TOOLING.md` for any other user who installs the plugin. A non-Vish user gets a rule that says their machine has a path it doesn't have. The 2026-05-15 patch added a disclaimer note ("adapt to your own machine") but the hard rule in the body still has the personal path. Because TOOLING.md is always silently overwritten, users can't even fix it permanently.

Scenario: any non-Vish user installs project-protocol on a new project; the agent reads `agents/TOOLING.md` and enforces a rule tied to a path that doesn't exist on their machine.

Direction: parametrize or remove the personal path from the rule body; the note is not sufficient.

**H2 — design-check auto-chain after build-component depends on the host environment firing it — this is not guaranteed**

File: `skills/build-component/references/phase-5-preview-and-write.md`, lines 146–158; `skills/design-check/SKILL.md`, lines 17–19.

Phase 5 writes the component and then says "design-check fires automatically" and "if the host environment doesn't auto-fire skills, surface the prompt." The auto-fire mechanism is description-match triggering in the host (Claude Code or Codex). But description-match triggering is non-deterministic: it fires when the model text matches trigger keywords in the description, not on a guaranteed post-write hook. The PostToolUse hook in `hooks.json` has no design-check dispatch — it only checks WORKLOG cleared state. So the chain is "happens to work when the model is lucky, silently drops otherwise."

Scenario: user runs build-component for a new card component; Phase 5 writes the file; design-check does NOT fire because the summary text ("wrote component") doesn't pattern-match "edit UI / change styles / add page"; raw hex values shipped with no scan.

Direction: add a PostToolUse hook (or a SubagentStop hook) that dispatches design-check after any Write to a known component path, or make build-component Phase 5 explicitly invoke design-check as a Skill call rather than relying on description-match.

**H3 — design-check trigger phrase list misses the most common real-world phrasing**

File: `skills/design-check/SKILL.md`, description field.

Direction: broaden the description trigger list to include "fix", "tweak", "update", "change the", "clean up", "adjust" when the context is a UI file; or escalate the PostToolUse hook approach.

**H4 — save-session Step 10 silently no-ops push on the most common solo-user workflow (working on `main` directly)**

File: `skills/save-session/SKILL.md`, lines 195–233.

Step 10c: "If current branch IS `main`: skip to 10d." Step 10d: "If current branch IS `main`: just run `git -C '<MAIN_PATH>' push origin main` and skip the merge." But this assumes `git worktree list --porcelain` returns a second worktree at `<MAIN_PATH>` that is different from the current directory. A solo user working in a single checkout (no worktrees) has only one worktree entry — the current directory IS `<MAIN_PATH>`. The `git -C "<MAIN_PATH>" push origin main` command runs in the same directory. This actually works if they're on main; it breaks when they have no remote (skip) or when the fast-forward fails silently. The deeper issue: the whole Step 10 mechanism was designed for Codex's worktree model. For Claude Code single-checkout sessions on `main`, the git flow degrades to a best-effort push with no validation that the merge path is even relevant.

Scenario: user has been running sessions on `main` branch without Codex worktrees; save-session's Step 10 either skips the push entirely (no remote) or runs a redundant push; there is no commit of the protocol .md changes if the git state happens to be clean at Step 10b check.

Direction: detect the no-worktree / single-checkout / main-branch case explicitly and reduce Step 10 to a simple `git add + git commit + git push` without the merge machinery.

**H6 — monorepo / nested-app detection is missing from TOOLING.md and phase-4 Node detection**

File: `skills/init-project/references/phase-4-design-system.md`, lines 108–122; `templates/TOOLING.md`.

Node detection is: "does `package.json` exist at the project root?" In a monorepo (`apps/web/`, `apps/api/`, root `package.json` is the workspace manifest), the root `package.json` triggers TOOLING.md installation, but TOOLING.md rules ("run bun dev", "one `next dev` at a time", "wipe `.next/dev`") target a Next.js app — not a workspace root. STRUCTURE.md detection in `build-component` phase-1 has explicit monorepo paths (`packages/ui/**`, `apps/web/src/**`) but phase-4 has no corresponding branching. Similarly, design-check's component glob paths assume a single `components/ui/` or `src/components/` at root — the monorepo paths are listed in build-component but not referenced in design-check's step-2 glob.

Scenario: user runs init-project on a Turborepo monorepo; TOOLING.md installs at workspace root and says "run bun dev" (not `cd apps/web && bun dev`); design-check later globs `components/ui/*` at repo root (finds nothing), skips reuse scan, proceeds.

Direction: add a monorepo mode to phase-4 node detection; gate TOOLING.md installation to the app-level `package.json`, not the workspace manifest.

**M1 — WORKLOG path detection in hooks.json uses a fragile env-var heuristic with no fallback for non-standard installs**

File: `hooks/hooks.json`, line 31; also lines 64, 75, 86.

The WORKLOG path is: if `${CLAUDE_PLUGIN_ROOT}` or `${CODEX_PLUGIN_ROOT}` is set → use `agents/WORKLOG.md`; else → use `cowork/WORKLOG.md`. This was fixed in v2.0.1 to make hooks work. But the heuristic assumes: Cowork = plugin-root vars not set; Claude Code/Codex = plugin-root vars set. If a user installs the plugin via a non-standard path (e.g., a symlink install or a future Cowork version that exposes plugin-root vars), the env-var check misfires and hooks write to the wrong WORKLOG. The PreToolUse "cleared state" check then either warns on the wrong file or silently no-ops.

Scenario: Cowork upgrades to expose `CLAUDE_PLUGIN_ROOT` in hooks; all hooks switch to `agents/WORKLOG.md`; Cowork sessions write discipline hooks to a file that Cowork agents don't read, making the whole hook layer invisible.

Direction: add an explicit session-type marker rather than relying on plugin-root env-var presence.

## 3. Cross-Cutting Observations

**The plugin assumes a single-app, single-checkout, macOS-bun-Next.js project.** At least 8 findings (H3 trigger gaps, H6 monorepo, H4 worktree, M3 Tauri, M6/M7 glob paths, M9 Cowork tool strip, L11 macOS path, TOOLING.md personal path) trace back to the plugin being designed against Vish's personal setup rather than the 15-project portfolio it's meant to serve. Any project that isn't a Next.js app on a Mac with a bun package manager and a single git checkout hits at least two of these gaps.

**Skill-to-skill chains rely on description-match, not explicit Skill tool calls.** H2, H3, and L9 are all instances of the same structural issue: the plugin documents skill chaining as if it's guaranteed, but the actual mechanism (description-match triggering in the host) is probabilistic. The only guaranteed execution path is explicit `Skill(...)` calls in the SKILL.md body. The plugin has zero explicit Skill calls between skills — every chain is implicit. This is the single highest-leverage architectural issue in the plugin.

**Undefined behavior accumulates in small defaults.** L2 (iteration cap conflict), L5 (template default treated as detected), M13 (canon protection ambiguity), M12 (discipline agent-self-invocation) are all cases where two reasonable agents would do different things. These "undefined agent defaults" don't surface in single-agent sessions but become visible across 15 projects with different agents and models.

**Swift / non-web projects are structurally unsupported, not just incomplete.** FUNDAMENTALS.md is entirely web-focused (React, Next.js, CSS variables, Tailwind, shadcn). `DesignTokens.swift` is named in design-check step 4 and build-component phase-5, but no template exists, no token chain is documented for Swift, and the DESIGN.md Extension protocol says "add to `DESIGN.md` AND `global.css` (or `DesignTokens.swift`)" without defining what `DesignTokens.swift` looks like. A Swift project using this plugin gets generic design-check passes and no concrete implementation path.

**The audit and design-check scans are detection-only with no memory.** Neither `audit` nor `design-check` stores findings between sessions. The same violation can be flagged in session 1, dismissed, and flagged again in session 2 with no history. The `DISCOVERIES.md` file exists for "what worked" but nothing records "what was deliberately left as-is." This produces noise in long-running projects.
