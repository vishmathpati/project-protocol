# Migration — v5.0.0

The Project Protocol operating-system redesign. This manifest is extended alongside the v5 implementation and must be applied transactionally by the final v5 migration engine.

## Phase 1 — Core operating system

### [RENAME] Core skills

- `session-recap` → `recap`
- `audit-before-close` → `completion-check`
- `audit` → `project-audit`
- `discipline` → `change-check`
- `discussion-mode` → `discuss`

Update root CLAUDE routing and any project documentation that explicitly invokes the old names. Historical changelog text is not rewritten.

### [MODIFY] Universal Foundation

Replace the old init-project mega-flow with repository inspection plus universal questions only. Never re-interview an initialized project. Never create real chapters, fake history, brand/design direction, marketing, or research during Foundation.

Always ensure root CLAUDE and the core brain files exist: `.plugin-version`, BRIEF, STATUS, STRUCTURE, WORKLOG, CHANGELOG, agenda, WONT-DO, and chapter guide/template. Preserve customized content and show semantic consolidation proposals for human review.

### [MODIFY] Role and chapter ownership

Worker authority becomes chapter-scoped inside an isolated worktree. Workers may propose any chapter-required file change. The CEO owns approval, reconciliation, and integration. Existing restrictive worker instructions in project CLAUDE files must be replaced after showing the diff.

Chapter templates gain Why, Method, Done When, relevant canon/dependencies, constraints, carry-over, Completion Reports, and Verdicts. Do not rewrite existing chapter history.

### [MODIFY] Session canon

- STATUS keeps current state, active chapters, blockers, pending human decisions, and one Last saved session.
- WORKLOG becomes a temporary meaningful recovery buffer; preserve unresolved entries.
- BRIEF becomes the current universal project contract; semantic consolidation is manual.
- CHANGELOG receives accepted or shipped outcomes only.
- agenda becomes the next human checkout/role/chapter/decision view.

### [REMOVE] `SITUATIONS.md`

Migrate unique routing knowledge into concise root CLAUDE routing, verify it, then remove the file. Never delete it before reconciliation.

### Safety

Do not stamp `brain/.plugin-version` as 5.0.0 until every mandatory v5 delta validates. Do not use `git add -A`. Do not touch project code. Preserve customized canon and unrelated dirty files.

## Phase 2 — Brand, marketing, research, and frontend execution

### [RENAME] UI research

Rename `calibrate` references to `ui-research`. Move `research_depth` out of DESIGN; ask it per UI Research invocation. Preserve existing concepts, teardowns, and screenshots. Normalize future output to `research/concepts.md`, `research/conventions.md`, `research/teardowns/`, `research/components/`, and `moodboard/manifest.md`.

### [REMOVE] Monolithic design direction

Retire `design-direction` routing only after mapping its responsibilities: BRAND diagnosis → Brand Foundation; research → UI Research; token alignment and preview → Style Lock; composition → Build Page/Component. Preserve existing BRAND, DESIGN, preview, research, and moodboard content.

### [MODIFY] BRAND and DESIGN

Preserve researched content. Add missing BRAND differentiation, emotional transformation, voice, cultural context, trust, memorability, distinctive/protected assets, reference tribe, and refusals. Convert project-wide DESIGN archetype to per-surface expression. Convert display-font bans to `avoid_by_default`. Never overwrite approved tokens automatically.

### [MODIFY] Marketing canon

Normalize CONTENT, SITEMAP, page briefs/copy, and MEDIA without forcing SaaS sections. Do not fabricate testimonials, proof, legal authority, or customer stories. Existing `marketing/layouts/` is legacy: stop consuming it, preserve or archive it, and migrate useful page-specific decisions into chapters only with human review.

### [ADD] Aside interfaces

Install/update standalone Aside `ui-research` and `inspect-component` skills when used. Screenshots remain local/gitignored by default unless redistribution rights are confirmed.

### [MODIFY] Frontend routing

Brand-facing substantial pages route through Build Page. Ordinary dashboard/product CRUD, settings, tables, and detail pages may follow existing project/shadcn patterns directly. Never auto-trigger UI Research or Inspect Component for ordinary dashboard work. Design Check runs preflight/postflight; Completion Check remains the contract gate.

## Phase 3 — Project Dashboard and hooks

### [ADD] Generated Project Dashboard

Offer deterministic generation of `brain/project-dashboard.html` from existing Markdown canon and local research assets. Tabs are Project, Brand, Design, Research/Moodboard, and Build Progress. Markdown remains authoritative; do not create a dashboard source file or selected-decisions database. Local relative assets are default; portable base64 embedding is explicit. Existing screenshots remain local/gitignored unless project policy and rights permit otherwise.

### [MODIFY] Hooks

Plugin-side hook updates take effect with the installed plugin. Remove old project guidance that promises per-action WORKLOG logging, pre-task NEW/ADDITION/UI/BUG classification, backup-file creation, SubagentStart/Stop entries, or automatic dashboard generation.

Keep only mechanical behavior: minimal Session Start role/Recap reminder and version drift; PreCompact branch/worktree plus useful WORKLOG tail; PostCompact same-role Recap routing; Stop warning on likely unpersisted state; deterministic post-write UI findings and Design Check routing; Style Lock synchronization reminder after DESIGN changes.

Hooks never select a role, approve work, run research, save canon, merge branches, or regenerate the dashboard.

## Phase 4 — Transactional migration and release integrity

### [MODIFY] Migration transaction

Migration first produces one consolidated read-only plan. Legacy layouts route through `migrate-to-brain`, which preserves README and project-specific rules, reconciles duplicates, never creates SITUATIONS, and never stamps an intermediate version. Deterministic deltas may be applied automatically; semantic changes require an approved diff. A skipped or unresolved mandatory delta leaves the migration incomplete.

The installed version is written only after target validation succeeds. Migration stages explicit owned files only and never uses `git add -A`, touches application code, switches worktrees, or merges worker branches.

### [MODIFY] Release audit

Replace stale prose/version greps and hardcoded skill counts with structural validation: manifest agreement, matching current migration, complete skill metadata, skill-name/folder consistency, hook schema and target existence, retired current-source detection, and packaging-debris checks.

### [MODIFY] OpenAI sidecars

Every shipped skill has display name, concise description, and a default prompt. Invocation policy remains explicit where user authority or destructive work is involved.

### Validation

Test read-only inspection against legacy v2.5, brain-based v3/v4.1, partial, and valid v5 fixtures. Planning must not mutate fixtures; incomplete targets must fail; a valid target must pass idempotently.

## Phase 5 — Integration closure

### [FIX] Worker authority contradiction

Remove any generated rule that limits workers to code plus the chapter file or says they may never touch shared canon. The governing rule is chapter-scoped authority: a worker may propose any chapter-required code or canon change in its isolated branch; CEO approval, merge, and shared-state reconciliation make it authoritative.

### [MODIFY] Metadata and routing parity

Codex sidecars use concise discovery descriptions and remain aligned with their SKILL contracts. Build Page describes the substantial brand-facing lane; ordinary product/dashboard work continues through established project and shadcn patterns unless a genuine missing component or explicit research need exists.

### Release gate

Run all skill validators, structural audit, migration fixtures, hook smoke tests, dashboard tests, JSON validation, retired-reference audit, and clean package inventory. Phase 5 introduces no new workflow group and no new project questions.
