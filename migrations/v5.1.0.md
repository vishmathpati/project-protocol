# Migration — v5.1.0

No mandatory project-file migration. Existing canon keeps working; the new contracts apply from the next authored artifact onward.

What changes for an initialized project:

- **Research packets.** New research runs produce `project-protocol.page-recommendations.v2` (job IDs, verbatim copy excerpts, `serves_jobs`, `first_impression`, media descriptions). Existing v1 packets remain valid: the dashboard and server accept both, and v1 renders exactly as before. Job↔copy pairing and staleness badges appear only for v2 data.
- **Content jobs and copy.** New or revised PAGE-BRIEFs record content jobs in the ID'd table (`<page-id>--job-<slug>`); new copy is written to `brain/marketing/copy/<page-or-family-id>.md` in the PAGE-COPY format. Existing free-text section intent and ad-hoc copy files are preserved and may be upgraded when their page is next touched — never rewritten automatically.
- **Stage B split.** Shared/site-wide copy locks after the approved site direction; each page's remaining copy is finalized inside that page's Build Page conversation. Projects mid-flight simply follow the split from their current position.
- **Approved Site Direction record.** New approvals may include shortlisted candidates, provided references, and focused-research outcomes. Existing locked records remain valid without these lines.
- **Style Lock and gates.** Research-led page builds now require locked design canon (Style Lock completed or an existing DESIGN explicitly reused) and Style Lock's preview is a throwaway style proof on real copy. Projects with an existing approved DESIGN satisfy the gate through explicit reuse.
- **Format stamps.** Templates now carry `> Format: project-protocol/<name>.v1`. Existing project files without a stamp are untouched; files gain the stamp when next created or intentionally migrated.

Nothing in this release rewrites BRAND, DESIGN, research evidence, decisions, copy, or page code automatically.
