# Migration — v4.1.0

The research-engine release. `calibrate` grows from a single moodboard-capture pass
into a **two-round, human-relayed design-research engine** driven by an Aside browser
agent: a SWEEP that maps the field into named concepts, a human checkpoint that picks
or blends one, then a DEEP TEARDOWN that forensically autopsies the chosen concept's
best real examples (real fonts, real palette, real motion stack). No behavior is forced
on existing projects — the new fields are additive and the engine only runs when
`design-direction` calls `calibrate`.

## [ADD] Install the Aside `design-research` skill (one-time, per user — not per project)

The engine's thinking runs inside the Aside browser. Install the static skill once:

- Source: `aside-skill/design-research/SKILL.md` in this plugin.
- Install it into your Aside skills (Aside → skills → add). It never changes per project;
  the per-project context arrives as a pasted mission prompt each round.
- If you do not use Aside, `calibrate` still runs via **Fallback A** (it drives Aside over
  MCP directly) or **Fallback C** (pure paste — you paste the summary blocks and calibrate
  reconstructs canon from them). No install required for the fallbacks.

## [ADD] `brain/BRAND.md` → `## Product` → **Niche** field

A one-line industry statement the sweep researches — e.g.
`- Niche: luxury hospitality — hotel, fine-dine, banquet, rooftop`.
Previously the niche lived only in prose; the research engine keys off it.

- Deep path: `design-direction` Phase 7 now writes it from the brand dump / axis extraction
  (`[VERIFY]` if auto-detected).
- Quick path: the `init-project` Phase 4 BRAND.md stub now includes it.
- **Existing projects:** add the line under `## Product` in `brain/BRAND.md` before running
  `calibrate`. If absent, `calibrate` will stop and ask for it (it never invents a niche).

## [ADD] `brain/DESIGN.md` frontmatter → **research_depth**

```yaml
research_depth: "[quick | standard | deep]"   # tunes Aside sweep appetite + teardown thoroughness; NOT a count
```

- `quick` — a fast, confident map of the obvious concepts.
- `standard` — the default; sweep to real saturation.
- `deep` — exhaustive; chase the long tail and edge variants.

Asked at `init-project` Phase 4 Step 0, beside the archetype question. **Existing projects:**
add the line beside `archetype:` in `brain/DESIGN.md` (default `standard` if unsure). It tunes
appetite only — it is never a site/concept/teardown count.

## [ADD] `brain/research/` — new research canon directory (created at research time)

`calibrate` (via Aside) writes:
- `brain/research/concepts.md` — the Round-1 field map (named concepts + evidence).
- `brain/research/teardowns/<slug>.md` — one forensic teardown per site of the chosen concept.

These are read by `design-direction` Phase 5 and by `build-page`'s external-reference
cross-check. No action needed to adopt — the directory is created when `calibrate` first runs.

## [MODIFY] `calibrate` is now a two-round orchestrator (behavioral)

`calibrate`'s outputs onto existing consumers are unchanged in **format**:
`brain/moodboard/notes.md` (same v4.0.0 per-entry shape, now citing teardown evidence) and the
`brain/DESIGN.md` FOLLOW/DEVIATE/REFUSE conventions audit (same block, now evidence-backed with
counts). Nothing downstream (`build-page`, `design-check`, `audit`) needs to change. The
governing law: **no hardcoded counts** — every quantity is saturation-driven.

## [MODIFY] `design-direction` Phase 4/5 wiring

Phase 4's handoff now describes the two-round sweep→concept→teardown pass; Phase 5 additionally
reads `brain/research/concepts.md` and `brain/research/teardowns/`. **Rule 7 preserved:** at least
one proposed direction must depart from every concept the sweep found — concepts map the field,
they never cap ambition.

## [MODIFY] `marketing-brief` — optional per-page convention check

When writing a page brief, `marketing-brief` may pose one light per-page convention question
(sourced from a targeted, saturation-driven Aside call scoped to a single page type). Optional;
no counts. This is where per-page conventions belong — at brief time, once the sitemap exists.

## [MODIFY] Root `CLAUDE.md` — drop the `edit-plugin` rows (cosmetic)

`edit-plugin` is now a workshop-only developer skill (moved out of the shipped `skills/` set), so
projects init'd on an older version carry two stale references to it. Removing them is optional and
purely cosmetic — the phantom rows do nothing if never invoked.

- Detection: root `CLAUDE.md` contains a skill-index row or situation-router row naming `edit-plugin`.
- Delta: delete the `| edit-plugin | … |` row from the skill-index table and the
  `| Editing this plugin's own skills… | edit-plugin | … |` row from the situation router.
- If absent: `[SKIP]`.
